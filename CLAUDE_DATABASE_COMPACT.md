# Claude Database Compact Rules

适用范围：数据库 Schema、ORM Model、Migration、SQL、数据治理项目的轻量规则。详细版参考 `CLAUDE_DATABASE.md`。

## 核心原则
- 你是资深数据库工程师，交付安全、可迁移、可回滚、可审计、可扩展的数据层设计。
- 质量优先级：数据正确性 > 安全性 > 可迁移性 > 可维护性 > 查询性能 > 极简。
- 修改前阅读现有 Schema、Migration、ORM Model、索引、约束、种子数据和相关查询。
- 遵循项目已有数据库类型、ORM、迁移工具、命名规范、索引策略和部署流程。
- 不写危险 SQL、不可回滚迁移、隐式破坏数据的脚本或无保护的大范围操作。

## Schema 设计
- 表名、字段名、索引名、约束名必须语义清晰，并遵循项目已有命名风格。
- 禁止用 `data`、`info`、`content`、`value` 承载核心业务含义，除非是明确 JSON 扩展字段。
- 主键、外键、唯一约束、默认值、可空性必须显式定义。
- 严格区分 `NULL`、空字符串、空数组、空对象的业务含义。
- 业务表应包含 `created_at`、`updated_at`；生产业务数据默认软删除。
- 金额、库存、计数、状态、权限、审计字段必须使用能表达业务约束的类型。

## 迁移规范
- Migration 必须可审查、可重复执行预期明确，并尽量可回滚。
- 禁止在单个迁移中混合大量无关 Schema 变更。
- 破坏性变更必须分阶段：新增兼容字段、双写或回填、切换读取、验证、删除旧字段。
- 大表新增非空字段必须提供默认值策略、回填策略或分批处理方案。
- 大表建索引必须考虑锁表风险，优先使用在线建索引能力。
- 数据回填必须分批、可中断、可重试，并记录进度。
- 禁止无条件大范围 `UPDATE`、`DELETE`、`TRUNCATE`，除非用户明确要求并提供保护条件。

## 查询、索引与事务
- 禁止拼接 SQL，必须使用参数化查询、Query Builder 或 ORM 安全 API。
- 查询必须明确字段选择，避免无必要 `SELECT *`。
- 列表查询必须分页并限制 page size 上限。
- WHERE、ORDER BY、GROUP BY、JOIN 字段应有合适索引；缺失时添加 `// TODO: Requires Index` 或迁移索引。
- 避免 N+1、无索引排序、隐式全表扫描、无边界聚合和大偏移分页。
- 关键查询应使用 EXPLAIN 或项目等价工具验证。
- 多表写入、余额、库存、权限、状态流转必须考虑事务、一致性和并发竞争。
- 并发写入必须考虑锁、唯一约束、幂等键、版本号或乐观锁。

## 安全、ORM 与交付
- 应用账号、迁移账号、只读账号应区分权限。
- 敏感字段必须考虑加密、脱敏、访问控制和审计。
- 日志、错误和调试输出不得泄露敏感数据、密钥或连接串。
- ORM Model、Entity、DTO 必须显式声明字段可空性，禁止让数据库 `NULL` 隐式穿透业务层。
- 数据库模型不应直接作为 API 响应，必须经过 DTO 或序列化层。
- Schema 变更后必须更新类型生成、DTO、校验器和相关测试。
- 完成后说明变更内容、影响表、索引变化、迁移风险、验证命令和回滚建议。
